# Family Wall Dashboard — Project Spec

A standalone, wall-mounted family dashboard built on Home Assistant. Self-contained (no phone required), touch-controlled, showing weather, calendar, live Reolink camera feeds, maps/traffic, and more. The Wall.

## Goals & Requirements

- Standalone: fully usable on its own — no phone needed to view or control.
- Surface: a large 43″ touchscreen in the living room / kitchen as the final build.
- Content: weather, calendar, live security camera feeds, maps/traffic, and other family info.
- Look: a custom, good-looking, app-like GUI — not the default HA web UI.
- Layout: a full-screen single view and multiple tiles, rearrangeable.
- Control: the family controls everything from the panel itself; a companion app is a bonus, not a requirement.
- Connectivity: pull Cat6 to the in-wall box now, while the wall is open. An in-wall, metal-backed location is poor for WiFi, and the electrician running power can drop the Cat6 at the same time. You can still start on WiFi (5GHz/WiFi6), but have the wired drop ready.

### Priorities (ranked)

1. Lowest cost / open-source
2. Ease of setup
3. Scalability / reliability
4. Ecosystem / integrations

## Core Platform: Home Assistant

Chosen. HA is the only option that delivers all the requirements locally and free:

- Native weather, calendar, maps, and live camera feeds (the deciding factor).
- Genuinely interactive on a touchscreen (not display-only).
- Mostly local and open-source — satisfies priority #1.

### Alternatives considered (and why not)

- DAKboard — polished and easy, but cloud-based with a subscription (~$5–6/mo), limited customization, weak on live camera feeds. Fails the open-source / low-cost priority.
- MagicMirror² — free and open-source, but display-only (no device control), landscape-first, weaker on live camera streaming and touch interaction.

### Known platform gap

HA cannot cleanly iframe Google Maps or Google Calendar's web UI (Google blocks iframing). Handled by the Earth/Maps kiosk-launch tile and the engine Calendar faces (which read the calendar API, not an iframe) — see tiles-spec.md.

## Hardware

### Server / "brains": Beelink EQi12 (Intel i5-1235U) — bought

- The box is a Beelink EQi12: Intel Core i5-1235U (10C/12T — 2 Performance + 8 Efficient cores, up to 4.4GHz), 32GB DDR4-3200, 512GB NVMe, Win11 Pro (wiped for HA OS). **Bought (~$470).** It supersedes the earlier EQ12 Pro (N305) plan — more machine on the two axes the Wall cares about most. On hand for the Phase-C soak test.
- Why this chip: the "buttery, no-jitter" bar is a GPU compositor job (an animated dashboard while WebRTC feeds decode on the same chip), and the deciding spec is iGPU execution units — the i5-1235U's Iris Xe carries **80 EU**, vs the N305's 32 and the N100's 24. The two real P-cores (vs the N305's all-E-core layout) keep HA's recorder/detection/automations off the animation thread, and Quick Sync v8 decodes HEVC/VP9/AV1 in hardware. More headroom on exactly the axes that buy smoothness.
- ⭐ Dual-channel RAM — verify before wiping Windows: Iris Xe only reaches full bandwidth in dual-channel; a single stick starves it (Intel even drops the branding to plain "UHD"). Confirm the 32GB is **2×16GB**, not 1×32 (Win11 → Task Manager → Performance → Memory → "Slots used: 2 of 2"). If it's a single stick, add a matched 32GB → 64GB dual-channel (the chip's max) or drop in a 2×16 kit. Dual-channel DDR4-3200 out-feeds the old N305 plan's single-channel DDR5, so once this checks out it's a non-issue.
- ⭐ Thermals — enclosure venting is load-bearing: the i5-1235U is 15W base but boosts to 55W, so it runs hotter under sustained load than the flat-15W N305 would have. Throttling = silently dropped frames. The EQi12 is actively cooled with an internal PSU, but the vented in-wall enclosure (top+bottom convection) is mandatory, and package temps get watched in the Phase-C soak.
- Storage — keep stock 500GB NVMe: the storage need is almost entirely camera recording, but the Reolink cams record to their own SD cards, so the box needs no 24/7 recording. HA + config + recorder DB + backups fit well under 100GB. Check the stock NVMe; swap to a quality 500GB (~$45) only if it's a DRAM-less QLC junk drive.
- Non-issues, logged so they don't get re-litigated: dual **Gigabit** LAN (not 2.5G) — camera substreams are single-digit Mbps each, so 1G is large overhead; Win11 Pro is bundled and wiped for HA OS bare-metal.
- Coral USB accelerator — dropped: detection (if Frigate is ever added) runs comfortably on the iGPU via OpenVINO at 2–4 cams.
- OS: Home Assistant OS bare-metal for least maintenance.
- Mounting: the box lives in a vented in-wall AV enclosure — see below.
- 2026 Beelink N305 refresh (EQ mini / EQi / ME Pro-2) noted but irrelevant: same N305-class chip, and it doesn't ship in configs/barebones until ~2027 — nothing to wait for.

### Display: 43″ 1080p PCAP touchscreen — TouchWo HD43, bought

- **Bought: the TouchWo HD43 ($916)** — a 43″ 1080p PCAP value panel, IPS/LCD (NOT OLED — see Burn-in). **Tradeoff accepted: it is NOT optically bonded and is air-gapped** (a small parallax/contrast cost vs a bonded panel, and a touch layer with an air gap to the LCD). The optically-bonded alternative was GeekLand GKTD43HDM (~$1,000, same Chinese ODM / identical touch module) — not taken; the TouchWo won on price + availability, and the air gap is an acceptable v1 compromise in a dark room. (The earlier spec carried the TouchWo as a $500 air-gapped *fallback*; the actual purchase came in at $916.) TouchWo includes a basic VESA bracket + the HDMI/USB cables.
- Resolution — 1080p based on price, 4K preferred if can be found near $1,000: tile views pull camera substreams (well under 1080p); full-screen 4K is marginal across a room; the camera's 4K value lives in its recordings + 5× optical zoom, not the wall; the Wall is big colour blocks, not dense text. The engine's stream-swap logic holds; only the cell pixel dims shift (a 1×1 cell on a 1920×1080 4×3 grid ≈ 480×360px, still smaller than the Fluent substream).
- Brightness — 300 nit is fine: the room is a dark living room (few windows). Lower brightness is better here — less glare at night, less burn-in stress.
- Touch — PCAP, NOT IR: >>>[OWNER]<<<'s #1 panel want is phone-like responsiveness. PCAP is phone touch tech; the chosen panels are optically bonded with a 200Hz touch scan rate (flagship-tier) and Mohs-7 glass. IR's hover-lag is rejected. Most of "phone feel" is won in software (the FLIP engine + the box's compositor headroom), not just the glass.
- Brightness control — software scrim, no hardware dimming: these value panels have no RS-232 / no documented DDC/CI, so the kiosk can't drive backlight brightness on a schedule. Overnight dimming = a software dark-scrim; screen-OFF still works via DPMS (standby <0.5W — the panel sleeps when the PC cuts the signal). (Worth a one-line email to GeekLand: does brightness adjust via DDC/CI over HDMI? Bonus if yes, fine if no.)
- Backlight life ~30,000h (budget-tier): ~3–5 years depending on daily on-time; the screen-off discipline stretches calendar life. A v1 panel, not a forever panel — fair at ~$1k.
- Market reality (why the value tier): a new commercial 43″ PCAP panel is a $2,000–4,000 product, often out of stock (GVision ~$2,300, Elo ~$1,840–2,590, faytech $4,000) — there's no cheap new 43″ touch because there's no consumer market for it. Premium zero-bezel 4K PCAP rejected on cost.
- Buy last: the panel is the most expensive item and the last needed (Phase B) — decoupled from the build. Go feature-complete on the Galaxy Tab + VM first; buy the glass when ready.
- Connection: HDMI (video) + USB (touch) from the mini PC.
- Keep the screen "dumb": don't rely on any built-in Android for compute — the brains are the box (EQi12). (Considered & rejected: making the panel itself a large Android tablet — large-format Android panels are signage-class that can't decode multiple 2K/4K Reolink streams, and hit end-of-support in 2–3 years.)

### In-wall mounting & enclosure

The panel sits slightly proud (~0.5–1.5″ off the wall, like a framed picture) on a thin low-profile VESA mount — NOT recessed-in-wall (structural framing + a heat trap). The box lives in a vented in-wall AV enclosure near the panel.

- Enclosure: top + bottom vents for convection (a sealed cavity throttles the box → dropped frames); separate line-voltage and low-voltage compartments (code); sized for the EQi12 (~5.0×5.0×1.7″) with generous airflow slack (the 1235U runs warm under load); serviceable by pulling the panel or offset just below it. Pick the specific box once wall depth (2×4 ≈ 3.5″, 2×6 ≈ 5.5″) + box position are known.
- Electrician work-order: a recessed 120V receptacle behind the panel (hidden) + a receptacle at the enclosure; spur off a nearby circuit (~120W total — no dedicated circuit needed). ⭐ Pull one Cat6 to the box now while the wall is open (home-run to the router/switch, terminate in an RJ45 keystone). Keep line/low-voltage physically separated per code. (Cat6 is low-voltage — >>>[OWNER]<<< or a handyman can pull it if the electrician won't.)
- Mount: GeekLand is 300×300 / M8 (needs a universal/adjustable mount); TouchWo includes a basic bracket (may want to upgrade for the flush look).
- Cabling: HDMI + USB box→panel, short run (USB ~16 ft passive limit). TouchWo includes cables; GeekLand buy.
- Power protection: >>>[OWNER]<<<'s whole-home surge protector covers surge — no UPS for v1 (HA is power-loss resilient + auto-backups exist; no room in the enclosure).
- On-site unknowns to resolve before the electrician: router/switch location (the Cat6 origin — the biggest cost variable) + exact box position relative to the panel (sets the HDMI/USB run length).
- Notification audio: both panels have built-in speakers fed over HDMI, but signage speakers are usually weak — bench-test; if chimes don't carry across the room, route through the HEOS system or add a small speaker.

## Kiosk & screen-off stack

A dumb 43″ panel driven by the box (EQi12), which runs HA OS/Linux — not Android. (This corrects the earlier Fully Kiosk / WallPanel plan — those are Android-tablet tools that assume the display is the computer; they can't run here.)

- Kiosk display — HAOSKiosk add-on is PLAN A: puterboy/HAOS-kiosk renders a full-screen kiosk straight over HDMI from HA OS (x86-tested), keeping the one-click add-on store and zero extra OS to maintain. **It runs OpenBox + luakit (a Lua-scriptable WebKit browser) under X, NOT Chromium** (correcting an earlier note). Verified alive + very active (v1.2.0+, Jan 2026): the REST API exposes `launch_url` (the shared launch-out primitive uses this), `display_on`/`display_off`, a settable screen-timeout, `refresh_browser`, and `run_command(s)`; plus configurable **gestures** (a corner-tap/swipe bound to built-ins like `kiosk.back` or `kiosk.launch_url`) and native long-press Back. **CORRECTION:** there is **no per-page CSS/JS injection seam** — the earlier "inject a styled back control onto Google's page" idea is not buildable here; the launcher's return is a kiosk gesture / native Back / an HA idle automation instead (see Open Questions + tiles-spec.md → Earth/Maps). (This corrects the old claim that HAOS can't drive a local display.)
- ⚠ **Single-touch only (Phase-C check).** The HAOS kernel ships without the HID multi-touch module, so HAOSKiosk gets single-touch input (the maintainer landed a PR adding it as a kernel module in newer HAOS — version-dependent). **Almost certainly fine for the Wall — every gesture is single-finger (tap, swipe, drag, pull-to-regenerate)** — but confirm on the panel that single-touch covers the gesture set, and check the installed HAOS build before relying on any two-finger gesture.
- ⚠ **Install watch (Phase C):** a recent add-on image-build error was reported on core-2026.4 (`InvalidDefaultArgInFrom` / base image). If the stable add-on won't build on the box, add the **testing branch** (`https://github.com/puterboy/HAOS-kiosk#testing`) as the repo URL.
- Fallback — TouchKio on bare-metal Debian (leukipp/touchkio): an Electron/Chromium kiosk that exposes screen on/off to HA via MQTT and gives full control of Chromium GPU flags. Use only if the animation needs GPU-flag control the add-on can't provide. VM work transfers either way (HA config is identical on HAOS vs Docker).
- Screen-off (load-bearing for burn-in) is an OS call: **DPMS standby** (NOT a hard signal-cut) to the panel (standby <0.5W), driven by an **HA schedule**. v1 has no motion sensor, so screen on/off is time-based — a manual override button lives in Quick Settings, and a future motion sensor just adds a presence trigger. DPMS standby (vs cutting the HDMI link entirely) is chosen so the **USB touch digitizer stays powered and a tap on the dark glass wakes the display** (the standard X11 input-wakes-DPMS path); the wake-tap is swallowed so it only wakes, never opens a tile. ⭐ Phase-C verify: confirm this value panel keeps touch USB powered during DPMS-off — if not, fall back to schedule-only wake + a physical button. Brightness dimming is a software dark-scrim, not ddcutil — the value panel has no RS-232/DDC. Screen-OFF is the load-bearing defense; scheduled backlight dimming is a nice-to-have covered by the scrim.
- Chromium GPU flags (--enable-gpu-rasterization, --ignore-gpu-blocklist, --enable-zero-copy) are a TouchKio-only concern — on Linux/Intel they can be the difference between composited-smooth and software-rendered-janky. Plan A (HAOSKiosk) is luakit/WebKit, not Chromium, so this isn't its decider; the plan-A-vs-fallback question is whether luakit composites the animation smoothly on the box (a Phase-C test on the panel). If it does, stay on Plan A; if the animation needs Chromium's GPU-flag control, fall back to TouchKio. Either way the wrapper's job is just "fullscreen, GPU-on, survives reboots, screen-off works"; the smoothness itself is won in the Wall's animation code (see engine-spec.md), not the wrapper.
- Lives in Phase C, not A: can't be prototyped in the MacBook VM (no real panel). Build/tune on the box.

## Burn-in Prevention (the full stack)

Primary defense is panel choice: LCD/IPS is far less susceptible than OLED — hence the IPS decision above. Layered mitigations, ordered by impact:

1. Scheduled screen-off + tap-to-wake (highest impact, load-bearing). The screen runs on a time schedule and goes truly off outside its window; a tap on the dark glass wakes it (DPMS standby keeps the touch digitizer live), with a manual override button in Quick Settings. v1 has no motion sensor — **motion-triggered wake is the documented upgrade path** (add a sensor → add a presence trigger to the screen-off automation). A screen that's actually off most of the day can't burn in and saves power. Mechanism: OS-level DPMS standby at the kiosk layer (panel standby <0.5W). Caution: must truly turn the screen off, not just dim a still-lit dashboard; verify the panel honors DPMS standby over HDMI. Not optional polish — the always-on tile surface depends on it. Built in Phase C (needs the real panel).
2. Cascade drift — the Wall slow-re-packs tile positions every ~30–60 min (activity-guarded), so no region stays lit the same way. This is the Wall's design contribution — full detail in gui-spec.md → Tile motion → Cascade drift. Camera caveat: a camera tile parked at 2×2 is the panel's worst case (four bright cells that don't move); cascade must move the whole 2×2 block as a unit, and screen-off must be airtight.
3. No permanent anchor — the Home tile drifts too, so even slot 1's pixels get relief.
4. Overnight dimming (software scrim) — the value panel may not have DDC, so there's potentially no scheduled backlight dimming; dim via a software dark-scrim if so. Lower stakes anyway (dark room + screen off overnight when idle).
5. Dark theme option — flat dark backgrounds, bold colour blocks; less static bright-pixel load.

Net: commercial IPS + scheduled screen-off (motion-wake a future add) makes burn-in effectively a non-issue. Screen-off is load-bearing, not optional — build it in Phase C, on the real panel.

## Scope & Audience: build for ourselves, share as a template (not a product)

Build v1 selfishly — tuned to our house, our cameras, our calendars, our taste, no compromises for hypothetical strangers. Build it cleanly (organized config, git repo, decisions documented across these specs) — cheap and good hygiene.

Explicitly rejected (for now): turning this into a turnkey, non-technical, public-installable product (guided onboarding, device auto-discovery, a settings UI instead of YAML, support for hardware/networks/calendars we've never seen). It's an enormous scope multiplier, much of it the wrong kind of work, and it would betray priority #1.

The realistic "public" path, if we ever want it: publish the config + a written guide as a template others can fork (the HACS/community norm). Costs a README, not a second product. Decide nothing on this until we've lived with v1 and still want to share.

## Notifications — the loudness policy (owned here)

A wall panel that cries wolf gets ignored, so very few events earn a screen-grab. Mapped onto the engaged/idle banner model — the banner's look is in gui-spec.md; which events are allowed to be loud is here. **Each source's tier is a per-source setting** (Admin → Events) with four choices — **Takeover / Banner / Log / Off** — backed by `input_select.wall_evt_*` helpers; the locked v1 defaults are below. The tiers:

- Takeover (auto-opens a tile + reflows when idle / banner when engaged — the screen changes). Every takeover follows the same rule: auto-open when idle, downgrade to a silent banner when engaged, audio muted in quiet hours. No per-event exceptions. The sanctioned list:
  - Doorbell press — someone physically at the door now.
  - Person, vehicle, OR pet on any camera — classified, never raw motion (pet is takeover by family choice — we like watching them; one toggle from banner). Vehicle is takeover-grade only because the detection zones are masked to exclude the road, so routine pass-by traffic never fires. ⭐ Guardrail: if a zone is ever widened, or a new camera is added unmasked, vehicle drops to banner for that camera — vehicle-takeover is predicated on the masking.
  - Severe-weather alert (a warning, not a watch) — the weather tile's red event-window.
  - Hurricane nearing — only when the computed closest-approach distance drops under a threshold (a storm tracking toward Texas does not fire); seasonal.
  - A further addition only gets in if >>>[OWNER]<<< approves.
- Banner (announce strip, no reflow — the workhorse tier): garage left open past a threshold (~10 min; silent banner first, chime only if still open after a second threshold); kitchen timer done (keeps its own classic sound); an irrigation "skipped for rain" / anomaly nudge. (A camera person/vehicle event arrives here only as the engaged-time or quiet-hours downgrade of its takeover, or as the guardrail fallback for an unmasked camera — not as a primary banner.)
- Log-only (records to the household log; no banner, no takeover) — the per-source "Log" choice, for sources worth a record without any on-screen interrupt (e.g. motion, if ever enabled). Distinct from Glance-only below (which is about tiles with no event behavior at all).
- Glance-only (no interrupt ever — lives on its tile): calendar events, markets, sports scores, trash-tomorrow, travel-time spikes. These already live permanently on their tiles; a push would duplicate them.

Two governing rules:

- Person, not motion. Every camera-driven interrupt requires a classification, never raw motion. The single most important tuning decision; raw-motion alerts are why people disable wall-panel notifications within a week.
- Quiet hours (tied to the overnight-dim schedule): takeovers downgrade to banners, banners go silent-visual-only. A 2am doorbell still flips the screen, just doesn't blast audio. Quiet hours has a manual **toggle** plus a configurable **time window** (`input_datetime.wall_quiet_start`/`_end`, default 22:00–07:00) — both Admin → Sound controls; the Mute button in Quick Settings (all-user, not gated) is the manual all-audio-off.

Audio (v1 = SILENT; receiver-audio parked to v2). v1 ships visual-only — bottom-strip banner + the household log, no chime system. The Wall has no dedicated speaker; the ONLY output path is powering the **Denon AVR** on and playing a tone/announcement through it — too slow + disruptive for instant events, defensible only for **sustained safety-grade events (severe weather, hurricane nearing)**. That, plus **TTS announce** and **STT voice control**, are parked to **v2**, to decide after living with the panel. Whatever audio v2 adds will still gate on Mute + quiet hours. (Mobile push: explicitly not wanted.)

(Two logs, two audiences. The **household notification history** — the catch-net for vanished doorbell/camera/weather/garage/sports events — lives in **Quick Settings** (the 1×2 menu). The **systems/maintenance history** — restarts, updates, backups, entity drops, integration re-auths — lives on the **Admin -> System** tab. **BUILT** this pass as ONE unified store (`www/diagnostics/diag.json`, written by `diag_append.py`) read by two filtered views: Quick Settings shows the household feed (camera-class events hidden) PLUS maintenance rows, the System tab shows the full `category:system` list. Schema `{ts, category, type, source, message, color}`; hardening later folds engine breadcrumbs (`category:diagnostic`) into the same pipe. Kitchen-timer-done stays out. Designs in gui-spec.md.)
