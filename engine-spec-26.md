# The Wall — Engine Spec

How the custom Lovelace card is built. This is the implementation source of truth for the-wall-card.js — the bespoke tile engine. It owns the how: FLIP math, the packer, the animation lock, the four-door layout as shipped, hass integration points, the layout model, and the build approach.

## Maintenance rules (read first)

- This file owns implementation only. FLIP/packing/layout mechanics, the four doors as built, hass wiring, motion-system internals, the build approach (custom card, no build step), and the engine punch list.
- It does NOT own design or features. gui-spec.md owns the GUI / interaction / motion design (look, behavior, motion intent, theme, the tile library, the Performance lens). family-dashboard-spec.md owns hardware, devices/integrations, features, build phases, the file manifest, and the entity reference. Reference those files by name; never restate their content here.
- Design wins conflicts. When design intent and the shipped code disagree, design is correct and this file (and the code) get corrected to match.
- This is a live snapshot, not a log. No session history, no "we changed X to Y," no change-log header. Describe only what the engine does now and what's planned next. Regenerate this file at session close-out: fold new implementation facts into the right section, move finished work out of "Not yet built," and delete anything no longer true.

- The dispatch "phone" (`tile._dispatch`) — the engine stamps a fire-and-forget `_dispatch(domain, service, data)` callback on every tile in `_buildOnce`. It's the ONLY way a face reaches hass, and it can only dial out (calls `this._hass.callService`, never reads). Keeps the renderer-never-touches-hass contract intact for WRITE faces. Reused by every control face. Surfaced to faces through the canonical tap contract as `ctx.dispatch`.
- THE CANONICAL TAP CONTRACT (the foundation pass). One mechanism for every interactive face — no face attaches its own event listeners. A face marks interactive elements with `data-tap-action="<name>"`; the engine's single delegated click handler, on a tap landing inside such an element, suppresses the generic step-back/step-deeper and calls the current face's exported `onTap(ctx)` hook. `ctx = { tile, action, hass, dispatch, el, refresh }`. The engine routes a named action to the owning face and knows nothing about what it means. This replaced climate's old `data-claims-tap` + per-button `addEventListener` wiring, so climate, radar, and (future) camera all obey one rule. A face action stamps `_lastTouch` (engagement).
- THE FACE-FIRST MODEL (the foundation refactor — see The four doors). Faces, not tiles, are the dispatch unit. A tile is a size ladder + per-size lists of faces BY NAME; the engine dispatches render/extract/onTap on the *currently-shown face's name*. The three maps (FACE_RENDERERS / LIVE_EXTRACTORS / FACE_TAP) are built from face descriptors `{ name, render, extract?, onTap? }` each module exports, keyed by face name. Per-face data: live data parks at `tile._data[faceName]` with a per-face signature (`tileId:faceName`), so a weather-sensor push never re-runs radar's manifest fetch and one face changing repaints only itself. Adding a face = write faces/<name>.js exporting a descriptor, list it in the FACES array in engine/face-maps.js, reference `{ face: "<name>" }` in a registry size list — zero engine edits, zero sibling-face edits. This is what makes the remaining ~13 tiles cheap. Migrated + verified via a node harness (built the maps, ran every extractor, rendered every face, fired both onTap hooks) before deploy.

No optimistic UI — a dropped Sensi change snaps back. Four named faces (climateTemp/Humidity/Control/Today) share one renderer (branches on face name) + one extractor. The Today face's trend line + run-bands read recorder history via the async seam (below); runtime total reads `sensor.climate_runtime_today`. THREE views (Greater >>>[HOME_METRO]<<< / Louisiana / Gulf Coast), tap-to-cycle via the tap contract (`data-tap-action="cycle-view"` → onTap bumps a module-level per-tile view index → `ctx.refresh()`). - **Cameras are LIVE (faces/camera.js) — the first LIVE-VIDEO faces, built on the engine's PERSISTENT VIDEO LAYER (no cold-start).** SIX face descriptors across three sizes: 1×1 flips cameraSingle ↔ cameraFrontDoor; 2×2 = cameraQuad (driveway + front-door live feeds top two quarters; bottom two = the activity pair — a snapshot reel + a filtered ScrollList); 4×3 = cameraFull ↔ cameraFullFront (the two live feeds) PLUS cameraSnapshots (the tap-only snapshot VIEWER, `swipeExclude`). **All live faces use the two Clear streams** (the Fluent split was dropped). A shared `singleFeedFace(name, stream, tag, class)` builder makes the four single-feed faces; the quad + the viewer are bespoke. **The live faces render NO video — only transparent `.cam-window` markers** (`data-cam-stream` / `data-cam-tag`); they opt into the engine's live-video machinery with `video:true` + `videoConfig(stream)` (the `custom:webrtc-camera` config, pointed at a go2rtc stream name; mount seam = `loadCardHelpers` → `createCardElement`). The engine owns ONE persistent per-tile video layer, keeps each stream's card warm + decoding, and repositions / FLIPs it over the markers — so flip/grow/reflow never cold-starts. The activity pair + viewer are STILLS (no decode), reading a local snapshot manifest via the `quadExtractor` (the radar/markets bake→read-local seam), and so never touch the video layer. Full mechanics in the LIVE-VIDEO seam + Persistent video layer (hass integration).
- **OPTION A — the live-element hass feed** (`_feedLiveEls`, called from `set hass`). A data face reads its values once and never touches hass again; a LIVE-VIDEO face embeds real HA cards that need a CONTINUOUS hass to keep streaming. Each mounted card is pushed into `tile._liveEls[]` (the engine does this for camera, in `_ensureVideoCard`; a face may for any other live element); on every push the engine re-hands the current hass to all of them, dropping the array when none are still connected (unmounted). Plus `tile._getHass()` (stamped in `_buildOnce` next to `_dispatch`) so a face can read hass ONCE at mount, before the next push lands. Narrow + reusable — the read-side twin of the `_dispatch` "phone"; the pattern any future live-HA-element face (live Maps, Media) reuses. A tile with no `_liveEls` is skipped for free, so the non-video tiles are untouched.
- **`ctx.openRung(rung, faceIndex?)`** — a new generic capability on the tap ctx: an interactive face can open ITS OWN tile to a specific ladder rung and optionally land on a specific face index. Cameras are the first user (a 2×2 quadrant carries `data-tap-action="open-feed"` + `data-feed-index`; its `onTap` calls `ctx.openRung(2, idx)` so tapping a quadrant opens 4×3 ON that feed). The faceIndex is landed BEFORE the reflow (inside `_openTile`), so the camera window-FLIP morphs straight to the tapped feed, not to face 0 then a snap. Lock-gated like any open; reusable for any "tap this part → open there" tile.
- **`ctx.flipTo(faceIndex)`** — a generic capability on the tap ctx: a face flips its OWN tile to another face on the SAME rung from inside an `onTap`, animated if the target is the adjacent face (reuses `_flipTile`), snapped otherwise. Markets is the first user (tap an overview-grid cell → `ctx.flipTo(0)` lands on the chart console, having re-aimed the chart at that ticker first); **Sports reuses it (built)** — an Overview cell tap selects that team + `flipTo(0)` to its Console. `flipTo` now delegates to a shared private `_flipToIndex(id, faceIndex, forceMotion?)` (extracted so the live-lock can reuse the same flip/snap logic; `forceMotion` pins the 180° page-turn for taps, omitted → a cycle tile crossfades).
- **PER-FACE TILE RECOLOUR — the `colorFor(data)` seam** (the `FACE_COLOR` map; faces may export `colorFor(data)`). The per-tile colour handles (`--tile-color` / `--tile-ink` / `--tag-color`) were set ONCE per tile from the registry `colorVar`; this seam lets a FACE re-point them from its own data when that face is showing. Factored `_setTileColorVars(node, colorVar)` out of `_buildTile`; `_applyFaceColor(id, faceIndex?)` resolves the effective colour (face `colorFor(data)` → per-entry `faceObj.colorVar` → tile `colorVar`) and is called at every visible-face-change chokepoint (`_repaintFace`, `_resizeFace`, `_flipTile` MID-MOTION — the colour swap is deferred to the point in each motion where it's visually hidden: a 3D flip at 90° (edge-on, neither face readable), a slide at its midpoint, a crossfade in lockstep with the fade (applying it at flip START snapped the background to the incoming colour while the OUTGOING face was still showing) — plus the `openRung` land, and build). `_setTileColorVars` treats `null` OR `--wall-neutral` as the neutral/charcoal case (tag → muted grey, since a dark-shade tag vanishes on charcoal). First user: **Sports' Saints faces** (`colorFor` → `--wall-neutral`, charcoal) — the 1×1 Saints glance always, and the Console when the selected team is the Saints (its `colorFor` reads the module's selected team, so a tap recolours immediately, not on the next push). This is the engine mechanism behind gui-spec's "event-override colour" — the doorbell/severe-weather overrides reuse it. A face with no `colorFor` never recolours (cheap no-op re-apply of the registry colour).
- **LIVE-LOCK — the `isPinned(data)` seam** (the `FACE_PIN` map; faces may export `isPinned(data)`). A flippable, closed, on-stage tile whose current face list contains a face reporting pinned is HELD on that face and excluded from the idle cycle, releasing when the predicate goes false. `_pinnedFaceIndex(tile)` returns the FIRST (lowest-index = highest-priority) pinned face index, or -1; a short pass in `_ambientTick` (after the engaged gate) snaps a pinned tile to its face via `_flipToIndex` and the cycle-gathering loop skips pinned tiles. First (only) user: **Sports** — a glance reporting `state === "IN"` pins the 1×1 cycle to the live game; multiple live → the fixed face order is the priority (FB > BSB > BKB > Saints). A manual swipe stamps `_lastTouch` → engaged → the pass pauses (you can browse other teams during a live game), re-pinning after the 2-min idle window. General primitive for any "hold this face while its data says so."
- **CONDITIONAL FACES — the `isLive(data)` predicate** (the `FACE_LIVE` map; faces may export `isLive(data)`). A face descriptor can carry an `isLive(data)` predicate; the engine builds a FACE_LIVE map and `facesFor(tile, sizeName, liveMap, data)` filters out any face whose predicate returns false, so a face appears in the rotation ONLY when its data says it should. Never empties a size (if every face filtered out, it falls back to the full list); `_repaintFace` clamps a stale face index when the list shrinks under it; `_allFaceNames` stays UNfiltered so extractors always run (a face must compute its data to know whether it's live). First user: the **Hurricane** faces (gated on `hasStorm` — present in the Weather rotation only during an active storm). Reusable for any seasonal/data-gated face (e.g. Sports in-season). A face with no `isLive` is always shown (the default).
- **SWIPE-EXCLUDE — the `swipeExclude` flag** (the `FACE_SWIPE_EXCLUDE` map; faces may export `swipeExclude:true`). A face so flagged is a **tap-only** face: `_flipTile` SKIPS it when stepping the swipe/auto-flip rotation (it steps past, wrapping, to the next non-excluded face), AND if the CURRENT face is swipe-excluded the flip is a hard no-op — so the face is reachable ONLY by an explicit `ctx.openRung(rung, faceIndex)` land, never swiped into, and it never auto-flips into view. Crucially the face STAYS in the size's face list (so `openRung`'s index land, which clamps against the FULL list, reaches it by true index) — only the swipe ROTATION walks around it. First (only) user: **Cameras' `cameraSnapshots`** — the 4×3 snapshot viewer, opened by tapping the 2×2 reel; it owns its OWN horizontal drag with a face-local **two-layer translateX slide** (a `.cam-viewer-stack` holding two `.cam-viewer-slide` layers — active visible, next pre-staged at ±100%; both translate in lockstep with the finger, on release past the engine's 36 px threshold the slide commits and the layers swap roles, else springs back; all transform-only) that mirrors the 1×1 video-face slide's feel verbatim (same `faceSwipeMinPx`/`swipeAngleMaxDeg`, drag-follow, click-swallow). It swallows the gesture so the engine's surface-swipe never sees it; the two engine guards above are belt-and-suspenders. The live feeds (`cameraFull`/`cameraFullFront`) thus flip only between themselves at 4×3, never into the viewer. A face with no flag participates in the cycle normally (the default).
- **SERVER-SIDE FETCH is the rule for external APIs (the CORS lesson — load-bearing).** A face must NEVER `fetch()` an external origin from the browser: the dashboard is served from `homeassistant.local:8123`, and any cross-origin `fetch` (NHC, IEM, SPC…) is blocked by CORS and dies silently (no error the user sees — just a face stuck "loading"). Radar's file-fetch works ONLY because it reads LOCAL files HA baked server-side — that pattern does NOT generalize to external APIs. The rule: HA fetches the external data server-side (a `rest:` sensor, or a `shell_command`/python writing to `/config/www`), exposes it as an entity or local file, and the face reads it synchronously like any other entity (`st`/`attr`) or via the local-file async seam. The Weather Station + Hurricane faces are the first to follow this — their IEM/NHC/SPC data all rides server-side `rest` sensors (see tiles-spec.md → Weather). Exception that's still fine: an `<img src>` to an external URL (images are NOT CORS-restricted, only `fetch`) — which is how the hurricane cone PNG loads directly.
- **IMAGE LAST-GOOD CACHE — the cone graceful-degrade pattern** (in faces/hurricane.js; reusable). For a face backed by an external image whose URL changes on a cadence (the NHC cone filename carries an advisory timestamp that rotates ~6h, while our sensor polls every 30 min — so the URL can briefly point at a just-rotated 404), the face keeps the last URL that actually fired `onload`, keyed by a stable id (the storm id). A pure `conePlan(data, cache)` decides what to show NOW: show the cached last-good while a background `new Image()` probe tests the candidate, promote the candidate to visible only on its `onload`, and on `onerror` keep showing the last-good — so an advisory-gap 404 never flashes a broken/empty box. Scope is keyed: the last-good only applies WITHIN the same id (a STORM CHANGE shows a "new storm · loading" note, never the previous storm's image under a new name). The probe is one preload per poll cycle — negligible. Reusable for radar or any rotating-filename external image.
- **The motion-notify hook** (`_notifyMotion` / `_isMoving` / the `FACE_MOTION` map; faces may export `onMotion(el, moving)`). Broadcasts each live face's own-tile motion state at every motion boundary (both reflow locks, flip start, all flip-settle exits, fresh paint). BUILT but currently UNUSED — it was added for the camera motion-freeze, which was then removed (see above). Harmless (empty map = no-op loop); left in place as the seam for any future face that genuinely needs to react to its tile moving. **Candidate to delete or keep-documented at a future cleanup.**
- ES-module split: the engine is a shell (the-wall-card.js) that imports per-face modules (faces/<face>.js) plus a shared helpers module (faces/_helpers.js) — native browser modules, no build step. Proven resolving in HA-served /www on the VM.
- **RESILIENCE — the per-face error boundary + the `_dispatch` wrapper + the diagnostics emitter (BUILT; Tier-1 hardening — the tier/policy detail lives in hardening-spec.md).** Every render funnels through `_safeRender(render, tile, faceObj)` — called by BOTH `_renderFace` and `_renderBoardAdmin`, the engine’s only two `FACE_RENDERERS[...]` call sites — and every extractor runs inside a try/catch in `_refreshLiveTiles`. So one throwing face or extractor degrades to an honest `_unavailableFace` (the `.t-*` glance grid) and the other tiles keep running, instead of one bad face blanking the whole Wall. `tile._dispatch` wraps its `hass.callService` in try/catch plus a promise-reject catch, so a renamed/deprecated service logs instead of stalling the interaction layer. All three paths log through `_emitDiag(type, source, message)` — a self-guarded (its own `callService` is try/caught and it never recurses), 5-min-deduped-per-`(type:source)` writer of a `category:"diagnostic"` row into the unified `diag.json` pipe (`face-error`/`extract-error`/`dispatch-error`, the first engine diagnostic emitters); the System-tab maintenance reader was widened to surface `diagnostic` rows. Validated (`node --check`/anchor/brace); the runtime catch is pending an on-glass injected-fault test.

## Build approach

The Wall is a custom Lovelace card >>>[OWNER]<<< authors — a Web Component (vanilla JS) that owns the entire DOM, CSS, and animation. HA hands the card device state via hass; the card renders the whole Wall — tile container, transforms, reflow — as its own HTML/CSS, while every HA integration (cameras, devices, automations) stays available for free. This is the only way to get the animation freedom the "buttery" bar needs.

- Hand-written, no build step — native browser ES modules. The card is split across files: the SHELL (the-wall-card.js) holds the Web Component + the four-door engine + TOKENS + TILE_REGISTRY + the built-in placeholder face; each LIVE face is its own file in faces/ exporting a `FACES` array of descriptors `{ name, render, extract?, onTap? }`; generic helpers shared by the shell and every face (MDI, st/attr/num) live in faces/_helpers.js. The shell imports each face module's `FACES` array and builds FACE_RENDERERS / LIVE_EXTRACTORS / FACE_TAP keyed by face name. No bundler, no tooling: the shell loads as a dashboard Resource of type JavaScript Module (so import resolves), relative imports resolve against /local/ (= config/www/), and the chat-only save-and-it's-live loop is preserved (drop file → /www, hard-refresh re-fetches the shell AND its imported face modules). A build step stays rejected — native ES modules give the file separation without one.
- **Claude Code carve-out (the ONE tooling exception, adopted at the UI-polish phase).** The standard loop is chat-only (>>>[ASSISTANT_NAME]<<< generates complete files → >>>[OWNER]<<< drops via Samba → hard-refresh). The agreed exception: when the project reaches the **UI-polish phase — rapid iteration on colour / layout / design changes to `the-wall-card.js`** (dropping many versions of the one shell file in quick succession), Claude Code (an agentic CLI that edits files in place) becomes the loop for that file ONLY, to skip the hand-copy friction. NOT for the normal feature loop. Same disciplines still apply on every write: `node --check`, exact-match edits, complete validated files. (Naming is exact: this is **Claude Code** for direct file editing — distinct from the HA MCP server, which is the voice-control future-want in family-dashboard-spec.md, a different tool for a different job.)

## Layout model — stage + scrolling drawer

The single fixed 4×3 grid is a stage (screen 0) sitting above a scrolling drawer. Layout is absolute px, not CSS grid.

- .wall is a scroll viewport (overflow-y:auto; scroll-snap-type:y mandatory; overscroll-behavior-y:contain; aspect-ratio:16/9, scrollbar hidden, touch-action:pan-y).
- .surface holds a pool of snap-spacer `<div class="snap">` (block flow, each --screenH = one viewport tall, scroll-snap-align:start) that set the scroll height + per-screen snap points, plus the tiles as absolute px children.
- _measure() is FIT-TO-WIDTH: stage width = viewport width, stage height is DERIVED (`vw * 9/16`), never read blind from clientHeight — so an off-ratio screen (16:10 Mac/tablet) gets the true 16:9 Wall with gutter letterbox bars top/bottom, and a wrong parent height can't mis-size every cell. A portrait-clamp fallback (if the derived height would overflow, fall back to fit-to-height) guards a tall/narrow screen; never triggers on landscape devices. It derives cellW/cellH and sets `--wall-unit` (= 1% of stage width) on `.wall` for the scaling system (see "Type + element scaling"). Recomputed by a ResizeObserver. _place(node,{screen,col,row,cols,rows}) writes the px box (a tile's top includes screen * viewportH). _updateSnaps(n) shows exactly _screenCount spacers. First layout is deferred to connectedCallback.
- The stage is the only place a tile is ever open, and is ALWAYS 12 cells full. Off-stage tiles are 1×1 and fill leftover stage cells first; the rest overflow to the drawer row-major + deterministic (never drifts under the user's scroll). With 12 tiles the drawer is ≤1 screen, so _screenCount ∈ {1, 2}.
- _scrollToScreen(i) smooth-scrolls; _freezeSnap/_thawSnap bracket reflows so flying tiles can't yank the viewport.

## Reflow — FLIP

_applyLayout(animate) implements FLIP (First-Last-Invert-Play):

1. First — capture every tile's current on-screen rect (getBoundingClientRect).
2. Last — write the new placement instantly via _place() (px left/top/width/height from measured cell geometry; no transition on these layout props). A drawer→stage move is just a large top change. The real layout is now final.
3. Invert — set each tile a transform: translate(dx,dy) scale(sx,sy) that visually puts it back where it was in step 1. The screen looks unchanged.
4. Play — on the next double-requestAnimationFrame, add a transition on transform and clear it to identity. The GPU animates each tile from its old apparent position to its new real cell.

Details that matter:

- Double rAF before Play so the inverted transform paints one frame before release (a single rAF occasionally skips the start of the animation).
- No-op skip: tiles whose dx/dy < 0.5px and sx/sy ≈ 1 are skipped — don't burn frames moving tiles that didn't move (this is why the deterministic drawer tiles don't animate during a stage-only reflow/cascade).
- Nothing hides. Every tile always has a position (stage or a drawer screen), so there is no display:none leave path; a tile that "leaves" the stage just animates to its drawer cell. The zero-rect fade-in branch is retained only for genuine first-paint.
- Scroll-snap is frozen during the reflow (_freezeSnap/_thawSnap); _updateSnaps runs before the first-rect capture so any scroll clamp from a shrinking drawer happens pre-capture (no FLIP jump). Thaw re-snaps to the boundary the user is already on.
- will-change: transform hints the compositor. backface-visibility: hidden lives on .tile-frame (not .tile); .tile is overflow: visible + perspective — required so the transient .flipper can do a real 3D flip without being flattened/clipped (see Flip mechanics).
- Content counter-scaling is applied: the inner .tile-face takes the inverse scale (1/sx, 1/sy) animating to identity in sync with the box, so text holds true proportions and never squashes mid-resize. Scale only is inverted, not translate; transform-origin: top left matches the box.

## The packer

_computeLayout() returns a Map of id → {screen, col, row, cols, rows} (0-based col/row; screen 0 = stage, screen ≥ 1 = drawer) for every tile. There are no null entries and nothing is ever hidden. Three parts:

1 — open tiles → the STAGE (mandatory), via backtracking (_solveOpen). Depth-first, largest-first (prunes fast), tries each tile's orientation options (flex) and candidate positions (_openCandidates), returns the placement Map if all open tiles fit on the 4×3 stage else null. The chosen span (post-flex) is written to spanOut and read back. Position-candidate ordering carries stability / nearest-fit / cascade:

- cascade mode (_cascading): randomized, avoiding the tile's last-1 cell (_posHistory) — drives burn-in drift on the stage.
- newest tile (_lastOpened): positions sorted by distance to its current stage cell — the nearest-fit "grows from your finger" rule (only meaningful for a stage→stage open; a drawer tile being opened has no stage cell yet and just takes first-fit).
- stable (other already-open tiles): current cell first — a no-move arrangement is found first so open tiles don't teleport.

2 — stage fillers (1×1). Remaining empty stage cells are filled with off-stage tiles from _offStageList() (= _order minus the open set; _order = registry order with any demoted tile floated to the front via _demote). Normally row-major; during a cascade, _driftAssign() scrambles the filler→cell assignment with last-1 avoidance so they drift too. The stage is therefore always exactly 12 cells full — open tiles plus enough 1×1 fillers.

3 — drawer (1×1). Whatever off-stage tiles are left after the stage is full go to drawer screens, row-major and deterministic — they never drift, so a tap-to-open never reshuffles the drawer under the user's scroll. _screenCount = 1 + ceil(drawerCount / 12).

Eviction = open-failure ONLY. If _openTile's trial solve returns null even with orientation-flex, it demotes the oldest OTHER open tile (LRU) to 1×1, calls _demote (floats it to the top of _order, so it re-seats as a stage filler if there's room) and retries. The demotion is committed to the LIVE _openIds set BEFORE the victim's _resizeFace repaint runs — otherwise _curSizeName still reads the victim as open and renders its OPEN face into the 1×1 cell (the bumped tile would keep its open layout crammed into 1×1 until the next flip). The just-opened tile is never the victim. A blank cell can never appear (there are always ≥ enough 1×1 tiles to fill the stage).

The trial-sizing rule (load-bearing). _openTile runs a trial solve before committing the new tile to the open set, so the solver must not read committed state for sizing. _openTile builds an explicit rungMap (id → intended rung for this trial) and passes it to _solveOpen; _intendedSizeName(id, rungMap) sizes every tile — including the not-yet-committed one — from the trial map. The solver writes the chosen span (post-orientation-flex) into spanOut, which _computeLayout reads, so geometry rendered == geometry packed (this is what prevents the blank/overflow cells a naive sizing-from-committed-state would cause).

Orientation-flex. sizeOptionsFor(tile, sizeName) emits the primary span plus its orientation alt (ORIENT_ALT: 1x2↔2x1, 1x3↔3x1); _solveOpen tries each option per tile and records the chosen one in _openSpanName so the layout + face render the orientation actually packed. This escapes the middle-row trap (a 2×1 sits in any single row). Opt out per-tile with orientLock:true — **Calendar is the first locked card** (its open week-box view only reads in 2×1 landscape; the packer is forbidden the portrait flip for it). A card's default orientation is still an open per-card design call.

Helpers: _freshGrid() / _canPlace() / _fill() operate an occupied[row][col] boolean grid (the stage); _cellOf(id) reads a tile's current stage cell from _curCell (returns null if the tile is in the drawer); _offStageList/_demote/_driftAssign manage the off-stage order; _shuffle() is Fisher-Yates. Packing decides geometry only; FLIP animates the move; _place() converts a {screen,col,row,cols,rows} to a px box. The three are deliberately separate.

## Geometry ceiling on 4×3 (the stage)

The binding constraint is the middle row, not total area. The grid has 3 rows, so any tile 2 rows tall (a 2×2 or a 1×2) must occupy 2 of the 3 rows — which always includes the middle row. The middle row has 4 cells; a 2×2 claims 2 of them, a 1×2 claims 1. So total middle-row demand of all tall tiles can never exceed 4:

- Max two 2×2. Three 2×2 is impossible despite 12-cell area = 12.
- Max four 1×2. Six 1×2 is impossible.
- 2×2 + two 1×2 fits (2+1+1 = 4); 2×2 + 2×2 + 1×2 does not (middle row already full); 2×2 + four 1×2 does not.
- 1×1s drop into the third row freely.

The packer returns null for the impossible sets, and _openTile then demotes the oldest open tile (LRU) so the new tile still opens — the user never sees a stuck state. The lever to escape the trap is orientation-flex (a tile rendered 2×1 sits in any single row). A taller grid (4 rows) is the other lever, but a bigger redesign. In the stage/drawer model this ceiling matters less than it used to — anything that won't fit just overflows to the drawer instead of being dropped — but the open set itself must still satisfy it to seat on the stage.

## The lock — one trigger at a time

_isAnimating is the load-bearing concurrency control. The reflow simultaneity rule (one tap or one event → one coordinated repack, not N independent ones) is enforced here.

- Set true when a reflow's Play phase starts; cleared by a timer at reflowMs + 30.
- Every trigger entry point (_onTileTap, _onEvent, _regenerate, _autoOpen/_onBannerTap) checks if (this._isAnimating) return; first, and settles any in-flight flip via _settleFlips() so a tap mid-flip leaves clean DOM before reflow.
- Proven under rapid tapping on the VM: mid-reflow taps are correctly ignored, no animation pile-up, no corrupted layout.

Gating adds a second gate (engagement) on top of this one (animation-in-progress); they are different concerns — do not collapse them.

## Triggers — the two ways a tile changes size

Maps directly onto gui-spec.md → "Three ways a tile changes size."

1. Tap (use mode) — _onTileTap(id): if the tile is open, _stepBack/_closeTile (removes it from _openIds); else _stepDeeper/_openTile (adds it, then resolves feasibility — see The packer). Open is additive + sticky: tapping a second tile opens it alongside the first.
   - Tapping a drawer (off-stage) tile PROMOTES it to the stage as a 1×1 — it does NOT open. _stepDeeper routes a closed in-drawer tile (_curCell.screen ≥ 1) to _promoteToStage, which floats it to the front of the off-stage list (_demote) so it takes a stage cell. If the stage has no free 1×1 cell (_stageFreeCells() < 1, where free = 12 − Σ open-tile areas via _curSpan), _promoteToStage shrinks the OLDEST open tile (LRU = _openIds insertion order) back to 1×1 — the same victim rule _openTile uses. Because _computeLayout re-solves the open set fresh and fills every freed stage cell from the front of the off-stage list, shrinking a larger tile (a 2×2 frees 3 net cells) automatically rides up the next drawer tiles in line; the stage stays 12 cells full. A second tap, now on-stage, opens it as usual (it FLIPs upward toward the stage; the viewport stays put and the user scrolls back up). Opening a tile already on the stage is unchanged.
   - Lock-gated; settles any in-flight flip first.
   - SEAM (camera + future 3-size tiles): when a tile is already open, deeper taps (which feed? full-screen? settings quadrant?) are intercepted inside that tile's face renderer before the generic close runs — the engine stays tile-agnostic and hardcodes no per-tile tap path. Documented in code at the _onTileTap open-state branch.
   - System gesture (not a size trigger): pull-to-regenerate. A gutter-only top pull (_installPull) calls _regenerate() — see Regenerate.
2. Event (system) — _onEvent(id): forceTakeover → always auto-open; engaged → banner; idle → _autoOpen (adds the target to the open set with the same eviction logic as a tap). _onBannerTap routes the same way.

Triggers 1 and 2 produce an identical end state (same open size, same reflow) — by design, so the user can't tell which path got them there.

## Size ladder, faces-by-size, colored frame

- Size ladder. _stepDeeper climbs a rung on a content tap; _stepBack (the colored .tile-frame band, or a content tap at the top rung) descends one rung; _closeTile at rung 0. Going back is always one rung down, never a jump to 1×1. Two per-tile registry flags refine the top rung (Home is the only user): **tapMaxRung** caps how far a CONTENT tap may climb — `_stepDeeper` clamps the effective top to `min(ladderTop, tapMaxRung)`, so any rung above it is reachable ONLY via `ctx.openRung` (Home's 4×3 Admin sits above `tapMaxRung:1`, so a Quick-Settings content tap can't climb in — the 7-tap gear's `openRung(2)` is the sole door); **borderBackOnly** makes a content tap at the effective top a no-op so only the colored frame steps back (Home's inert Quick-Settings content — notification rows, gaps — neither closes nor drills). Most tiles are 2-rung (1x1 → open); Cameras is the 3-rung case (1x1 → 2x2 → 4x3). Max 3 rungs. **4×3 STAGE-RESTORE (BUILT):** opening any tile to a 4×3 takeover consumes all 12 cells, so `_openTile` evicts every other open tile — which would leave an empty stage when you back out. The fix: at the TOP of `_openTile`, the first time a `4x3` span opens with no snapshot pending, capture the ENTIRE open stage (every open tile's rung INCLUDING the takeover tile's own prior rung, read before the overwrite; rung 0 = was-closed) into `this._stageSnap` (+ `_stageSnapFor` = the takeover id). `_stepBack` from that 4×3 calls `_restoreStage()`, which rebuilds `_openIds`+`_openRung` from the snapshot in ONE pass (`_resizeFace` each, a single `_applyLayout(true)`) → a clean reverse animation back to the prior multi-tile stage. `_closeTile` drops a pending snapshot if the 4×3 is closed directly; `_regenerate` clears it. State inits (`_stageSnap`/`_stageSnapFor` = null) live in the core constructor beside the open maps. Rung tracked in _openRung; the packed orientation in _openSpanName. _curSizeName prefers the packed orientation so face + frame match what's on screen; _resizeFace/_renderFace re-render + reseed flip eligibility on every rung change.
- Faces by size. facesBySize[size] overrides the default faces list per rung; facesFor(tile,size) resolves with fallback to the closed list — so you only declare overrides where a larger size genuinely shows different content. _renderFace passes (tile, faceIndex, faceObj) so a renderer can branch on faceObj.k.
- Colored frame = identity + universal back-button. Each tile wraps its face in .tile-frame (category-colour band = identity + back target at open sizes; thin/decorative at the closed 1×1; data-open toggles the band; fixed-px band width so all open tiles match regardless of width). The band's hit zone is wider than the visible colour (mis-tap guard). At full-screen 4×3 there is no gutter, so the frame is the only way out.
- Tap contract seam (canonical). A face declares interactive elements with `data-tap-action="<name>"` and exports an `onTap(ctx)` hook; the engine's delegated click handler routes a tap inside such an element to `_onFaceAction`, which suppresses the generic step-back and calls the current face's `onTap` with `{ tile, action, hass, dispatch, el, refresh, openRung, flipTo }`. No face attaches its own listeners. First users: climate's Control face (setpoint ±/mode/fan → `ctx.dispatch`) and radar's view-cycle (`cycle-view` → bump view index → `ctx.refresh`). The camera's pause/PTZ at the top rung will use the same contract.

## Regenerate + the scroll model

The all-1×1 selector (collapse / restore / snapshot / edge-swipes) is gone. Its two jobs are served by the always-present drawer (you can already see every tile — scroll down) and a single pull-to-regenerate gesture (one tap to reset the deck).

- _installPull(root) — pull-to-regenerate (gutter-only, top-only). On pointerdown: bail if scrollTop > 1 (must be at the very top) or if the target is inside a .tile (must start in the gutter / dark background, not on a card). On pointermove down: damped rubber-band (dy * pullDampen, capped at pullMaxPx) translating .surface; captures the pointer and preventDefaults. On release past pullTriggerPx → sets the click-suppress window and calls _regenerate(); otherwise springs back. The bottom edge does nothing (it's just the end of the drawer scroll). Single-pointer.
- _regenerate() — settle any flip, clear _openIds/_openRung/_openSpanName, _lastOpened = null, reset _order to registry order, re-render every tile to its closed face 0 (_resizeFace), clear the pull transform, scrollTo({top:0}), then _applyLayout(true). Result: all 12 tiles as 1×1 on the stage, drawer empty, view at the top.
- Stage return. _ambientTick calls _scrollToScreen(0) once the view is scrolled into the drawer and idle ≥ openTileIdleMs — a panel left mid-scroll drifts back to the stage on its own.

## Flip mechanics — true two-faced flip + delegated swipe

True 180° flip (_doFlip3D / _finishFlip). On a flip the engine builds a transient .flipper (position:absolute; inset:0; transform-style:preserve-3d) inside the tile, moves the current .tile-frame into it as the front, and adds a new .tile-frame (rendered for the next face) as the back, pre-rotated rotate{Y|X}(180deg) so it lands upright. The flipper animates rotate{axis}(0 → ±180deg) in the swipe direction; backface-visibility:hidden on each frame shows only the side facing you; the tile's own background:var(--tile-color) covers the brief edge-on moment. _finishFlip promotes the back frame to be the tile's sole .tile-frame, removes the flipper, advances _faceIndex. Transform-only → GPU-cheap.

- This is why the CSS moves backface-visibility onto .tile-frame and makes .tile overflow:visible + perspective (an overflow:hidden element is forced to flatten its 3D children). The .tile-frame keeps overflow:hidden, so content clipping is unchanged. The steady-state DOM is identical to before a flip, so reflow / _repaintFace / the data-open band CSS all keep working.
- Manual swipe always uses this on axis y in the swipe direction (_rotateFace → {kind:'flip',axis:'y'}). Auto idle-flip's _pickFlipMotion returns {flip, axis:'y'|'x'} or {slide,dir}; cycle-mode tiles still crossfade. _settleFlips finalizes a 3D flip (via _finishFlip) or cleans a slide/cycle, so any trigger can interrupt mid-flip safely. Per-flip timeouts live in _flipTimers.

Delegated face swipe (_installSurfaceSwipe). One handler on .surface, not per tile. pointerdown records the origin tile from e.target.closest('.tile') (a gutter start → no swipe, the pull owns it); pointerup decides from the start→end vector — |dx| ≥ faceSwipeMinPx and angle ≤ swipeAngleMaxDeg → _rotateFace(id, dx<0?+1:-1) + set the click-suppress window. Because tracking lives on the shared parent, a swipe that crosses the tile boundary still completes (a per-tile pointerup never fired off-bounds). If the finger releases over a different tile, the synthesized click fires on the common ancestor (not a tile) so it can't open anything; if it releases on the same tile, the suppress window blocks that click. No per-tile pointer-capture (that approach regressed before). Vertical drags fall through to native scroll (touch-action:pan-y).

- Swipe-angle gate. The angle test is Math.atan2(Math.abs(dy), Math.abs(dx)), which measures deviation-from-horizontal in [0°, 90°] regardless of direction — so both a left and a right swipe pass the ≤ swipeAngleMaxDeg test and flip (turning opposite ways). Using the raw vector angle (Math.abs(Math.atan2(dy, dx))) is wrong: a leftward swipe reads ~180°, fails the test, falls through, and the synthesized click opens the tile — the asymmetric "right-to-left opens instead of flips" bug.

## Board flip — the Control Room 3D reveal (Stage 2, BUILT)

A WALL-LEVEL flip, distinct from the per-tile flip above — the whole stage turns to reveal the Control Room on its back. The 7-tap gear's `onTap` calls `ctx.flipBoard()` (the `ctx` seam added in interactions.js) → `_flipBoard()` (motion.js). The live `.surface` subtree REPARENTS (no teardown — tiles, the persistent video layer, and scroll position all preserved) into a transient `.board` (the only Wall-level `transform-style:preserve-3d` element, a child of `.wall`): the live stage rides `.board-front`, the Control Room rides `.board-back` (pre-rotated `rotateY(180deg)` so it lands upright, `backface-visibility:hidden` on each face). `.board` animates `rotateY(0 → 180deg)` over `TOKENS.boardFlip.durMs`; `.wall` gets `perspective` only while the `board-flipping` class is on. The colour-border back-band (`.board-back-band`, accent handed in from the Home tile's `--tile-color` — single-source, no hex) IS the flip-home button. `_unflipBoard()` turns back to 0deg, then `_boardSettle()` reparents `.surface` to `.wall` and drops the board — the steady-state DOM (`.wall > .surface`) is restored, so the packer / reflow / video layer never see the board. Transform-only → GPU compositor.

- **The CR back-face is rendered by `_renderBoardAdmin()` (core.js) reusing the registered `homeAdmin` renderer VERBATIM**, so the board CR is byte-identical to the 4×3 Admin rung, and the `homeAdmin` extractor keeps `tile._data.homeAdmin` fresh. `_repaintBoardBack(force)` (called from `set hass`) re-renders it in place WITHOUT turning the board — **signature-gated** (only when the homeAdmin data moved; HA pushes hass constantly, and an ungated rebuild tore the CR DOM down on every push — the lag + "works once then dies" bug). A FORCED call (a CR tab/arm tap, whose state isn't in the hass signature) always rebuilds.
- **Ambient is frozen while flipped** — `_ambientTick` early-returns on `_boardFlipped || _boardFlipping`, and `_refreshLiveTiles` skips the hidden front-stage entirely (only the back-face refreshes). `ctx.refresh` from a CR control routes `_repaintFace('home')`, which DELEGATES to `_repaintBoardBack(true)` while flipped (else it would repaint the hidden front tile and the CR would look dead).
- **CR-internal taps** are wired via a click listener on `.board-back-band` (the board-back lives OUTSIDE `.surface`, so the per-tile listener never sees it): a tap on a `data-tap-action` element routes through `_onFaceAction('home', …)`; a tap on the BORDER (outside `.board-back-inner`) fires `board-home` → unflip; a tap on **blank CR space is inert** (the band no longer self-carries `board-home` — >>>[OWNER]<<<'s ask: back is border-only).
- **Two gotchas fixed on glass (both pre-existing fragilities the board exposed):** (1) **pull-to-regenerate** treated a scroll-release inside the CR as a gutter pull and bounced the whole stage — because the CR rides `.board` (OUTSIDE `.tile`) and the pull handler only excluded `.tile`. Fix: the pull bails on `_boardFlipped`/`_boardFlipping` or any pointerdown inside `.board`. Every other ScrollList was safe only because it lived inside a `.tile`. (2) A `wireScrollList` drag inside the CR could be rebuilt out from under the captured pointer — fixed by a `_boardBackBusy` flag (set on pointerdown inside `.board-back-inner`, cleared on up/cancel) that skips the gated rebuild mid-scroll.
- An MDI icon coloured via a parent inline `color` stays the stylesheet colour, because `.cr-sec-extra .mdi { color: var(--cr-accent) }` targets the icon directly — override by setting `--cr-accent` on the wrapper, not `color` (the offline-count red-icon fix).
- Files: `tokens.js` (the `boardFlip` block), `styles.js` (`.board`/`.board-back`/`.board-back-band` + the `.wall.board-flipping` perspective), `motion.js` (`_flipBoard`/`_unflipBoard`/`_boardSettle`/`_buildBoardBack` + the ambient freeze + the busy guard), `core.js` (board state, `_renderBoardAdmin`, `_repaintBoardBack`, the `_repaintFace` delegate + the `_refreshLiveTiles` skip), `interactions.js` (`ctx.flipBoard`/`unflipBoard` + the pull bail), `faces/home.js` (gear → `ctx.flipBoard()`; the `board-home` onTap case).

## Motion systems — as built

From gui-spec.md → Tile motion. All built and proven on the VM. Performance discipline (the repaint-one-face rule, stagger-as-frame-budget-tool) is implementation; the Performance lens framing lives in gui-spec.md.

- L1 — idle/engaged gating. _isEngaged() (last-touch within ambientIdleMs OR simulated presence) is the single gate for all unprompted motion. Reflow is NOT gated (it is the system responding to a trigger, not moving on its own). Events downgrade: idle → auto-open + reflow; engaged → banner; severe-weather (forceTakeover) → always takes over. The banner mechanism is a **bottom-anchored, STACKING** strip set (newest enters at the bottom gutter and pushes the stack up; oldest at the top fades first; capped at 3 visible; settles on expiry), tap-to-switch, ages out per `wall_banner_tail_s`. It is a **Wall-level overlay layer that floats ABOVE the grid and never reflows or repositions tiles** (the same overlay-plane idea as the PTZ seam, but Wall-wide rather than per-tile). Its element, the engaged→banner downgrade branch, the stacking + overlay-float, AND the real-HA-event wiring are **BUILT + live**. `_scanEvents(hass)` seeds a baseline on the first push then edge-detects 14 real entities → 9 logical sources each turn; per-source `input_select.wall_evt_*` policy gates each to takeover/banner/log/off. On a **takeover** the engine opens the target tile to a **specific rung+face** (`_takeoverRung` — cameras 4×3 follow the tripped cam [front-door→`cameraFullFront`, else `cameraFull`]; severe→`radar`, hurricane→`hurricaneOpen` cone, sports→`sportsConsole`, rain→`radar`; face resolved by NAME→index so a reordered list can't break it) and fires a **trimmed label banner** (`takeoverBannerMs`, ~6s, no arrow, tap-to-dismiss); the **engaged** path keeps the full ~20s tap-to-open strip. NOTE: the hurricane face is `isLive`-gated on `hasStorm`, so a hurricane takeover lands on the cone only when a storm is actually active (the only time the event fires) — with no storm it falls back to a weather face (expected, not a bug).
- L2 — flip-to-reveal. Per-tile content motion in place, idle-only, gated. Each flippable tile owns a randomized next-flip time (desync). Motions mixed per flip on flip-mode tiles: H-flip (rotateY), V-flip (rotateX), slide (push, random edge); cycle-mode tiles crossfade. Concurrency-capped (maxConcurrentFlips), most-overdue-first (no starvation). Auto-flips fire only for on-stage tiles (drawer tiles, usually scrolled off, don't burn the flip budget). Open tiles don't auto-flip.
- L3 — cascade drift. Slow whole-stage position re-pack for burn-in (cascadeEveryMs), gated by _cascadeAllowed() (any open tile earns the longer openTileIdleMs grace). During a cascade the position-candidate ordering switches to randomized + last-1-cell-avoiding (_recordHistory/_posHistory) for both open and filler tiles, so nothing ping-pongs; _driftAssign scrambles the 1×1 stage fillers. Open tiles drift too but are solved first so they can never be hidden. Re-packs the stage only. Manual shuffle() trigger (Settings control / burn-in reset).
- Counter-scaling. Inner face takes the inverse scale during reflow (see Reflow). Verified by math; the first real face (Weather) is the visual confirmation.
- L4 — system choreography. Center-out wake entrance (_playEntrance(), console-replayable; the real DPMS/kiosk wake hook is Phase C). Animates the stage tiles; drawer tiles sit out.

Settled placement/variety calls (the alternatives were tested and dropped): open placement = nearest-fit (topleft/anchor dropped — anchor is a strict subset of nearest); flip variety = H-flip / V-flip / slide (a true diagonal-axis flip is geometrically impossible as a clean slab — it reads as a wobble; slide is the authentic Metro third motion); entrance = center-out (reading + edgeslide dropped).

## hass integration

- As built. set hass(hass) stores this._hass and calls _refreshLiveTiles(). That walks the registry; for each tile it runs EACH of the tile's live faces' extractors (the only place entity IDs live — collected once via `_allFaceNames`, the distinct face names across all the tile's sizes), builds a small plain data object per face, compares a per-face JSON.stringify signature (`tileId:faceName`) against _dataSig, and on a real change parks the data at `tile._data[faceName]` and (if the currently-visible face changed) calls _repaintFace(id). _repaintFace rebuilds only that tile's currently-visible face node — never the grid, never another tile (structurally enforces the top live-build perf risk). If a repaint can't happen cleanly (mid-reflow _isAnimating, mid-flip _flippingNow, tile hidden) it defers by deleting the CURRENT face's `tileId:faceName` _dataSig entry, so the next hass push re-detects and retries — HA pushes state every few seconds regardless of engagement, so a live tile keeps updating even while someone is at the panel, with no ambient-tick coupling.

  - **Resize = window FLIP.** In `_applyLayout`, BEFORE each tile's inverse transform is set, `_collectVideoFlip` measures each warm window's NEW marker rect in the clean pre-transform frame and FLIPs the window from its captured pre-jump rect to that target with its OWN transform (a composited `<video>` does NOT honor an ancestor transform, so the window needs its own), in lockstep with the tile reflow; appearing feeds crossfade in at their new rect, leaving feeds crossfade out. Type + element scaling — the `--wall-unit` system (the fit-to-width scaler). One root unit governs every screen-consistent size, so the Wall looks identical in PROPORTION on any screen (the Mac, the tablet, the 43″ panel) and only the absolute px tracks the screen. Two scaling bases, deliberately distinct:
- **Scale to the SCREEN** (same physical size on every tile + every screen): the tag, glyph, detail, rain line, face padding, all radar/weekly overlays, and every `_components.js` atom. The multipliers are unitless and live in `TOKENS.scale` (the revived, formerly-dead type-size block), injected once as `--u-*` CSS vars on `.wall` by `_injectScaleVars()`. This replaced the 40+ per-element `clamp(min, Ncqw, max)` triples — dialing the whole Wall's look is now one table.
- **Scale to the TILE** (intentionally bigger on a bigger tile): the hero number, climate's "now"/setpoint numbers. These KEEP `cqw` (container queries; `.tile` is `container-type: size`), capped by `--u-hero-max` as the only safety rail (stops cartoonish growth on an ultrawide). `cqw` was NOT killed everywhere — only where the goal was screen-consistency.

ScrollList — CANONICAL scrolling-list component (**BUILT** in `_components.js` — a `ScrollList({rows,empty,klass})` string atom + a `wireScrollList(el)` initializer for the affordance state; CSS in the shell `_css()`. **Calendar is the 1st adopter — it built the primitive** [its 1×1 week-list + open day-detail]; **Cameras** [2×2 event list] and **Media** [Movies open list-block] adopt it next). A reusable contained-scroll list primitive — like the `.w-*` atoms, defined once and adopted by any face that needs a scrollable list. Behavior baked in:
- **Contained vertical scroll** that coexists with the Wall's gestures. The baseline is light: the engine's resize-all-to-1×1 vertical swipe arms **only when the drag originates in the top gutter**, so a vertical drag inside a tile body doesn't hit it, and the list mainly has to stay orthogonal to the HORIZONTAL swipe-to-rotate-faces. Requirement: a vertical-intent drag scrolls the list; a horizontal-intent drag passes through to rotate the face. **One exception now exists:** a face MAY itself claim vertical drags for paging (Calendar's open week-box pages weeks on a vertical swipe — NOT a ScrollList, a fixed-box slide-pager). When a face owns vertical paging it does NOT also host a vertical ScrollList in the same region — the two never share an axis on one face. So the rule generalizes to: per face, vertical is owned by exactly one consumer (the drawer-resize gutter swipe, a ScrollList, or a face's own pager), resolved by where the drag starts + which face is open.
- **State-swapped affordances (mutually exclusive — never both on screen, which keeps a ~480×360 quadrant uncluttered):**
  - *At rest:* directional chevrons show, appropriate to scroll position — down-only at top, up-only at bottom, both mid-list, NEITHER if the content doesn't overflow. Scrollbar thumb hidden.
  - *While scrolling:* the scrollbar thumb appears for live drag position (proportional — a short thumb = lots off-screen). Chevrons hidden.
  - *Scroll ends:* thumb fades after a short idle-timeout (~400ms); chevrons fade back in.
  - No overflow → neither affordance ever.
- **Scroll-active detector:** a scroll-event listener + ~400ms idle-timeout drives the at-rest↔scrolling state swap. Stateful face logic, lives in the SAME touch seam as the vertical-scroll-vs-horizontal-swipe disambiguation above — both built together.
- **Empty state:** renders a clean empty body when the backing list is empty (the empty text is the adopter's to supply).
- **Pointer-capture drag + inertia (BUILT — the hardening that closed the Calendar 1×1 bug).** `wireScrollList` now OWNS the vertical drag the way the surface owns the horizontal swipe: the `.w-scroll-body` is `touch-action:none`, and a `pointerdown` is axis-routed on the first move — a vertical-intent drag `setPointerCapture`s the body and drives `scrollTop` directly (so a drag that STARTS inside the list still registers when the finger finishes OUTSIDE it), a horizontal-intent drag is released untouched so the surface swipe still flips the tile, and a sub-threshold press stays a tap (the row's `data-tap-action` routes normally). On release a flick coasts with velocity-decay momentum; a `free`-mode list (the `w-scroll--free` flag) adds a soft transform end-bounce, `contained` (default) hard-clamps. A one-shot click-swallow after a drag stops a flick that ends on a row from also firing that row's tap. This **fixed the old closed-1×1 bug**: the list no longer needs a definite flex chain to bound (the JS clamps `scrollTop` itself), and the outer `.wall` `scroll-snap` can no longer hijack a 1×1's inner scroll (the prior root cause). The affordance state (chevrons at rest / thumb while scrolling, the ~400ms idle fade, no-overflow) is unchanged — programmatic `scrollTop` changes fire the same `scroll` event, so the thumb tracks the JS-driven scroll for free.
