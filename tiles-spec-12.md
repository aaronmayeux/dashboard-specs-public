# The Wall — Tile Implementation Spec

The wiring behind each tile: which integration backs it, the entity IDs, the data-source specifics and gotchas, and what's actually built. This is the source of truth for how a tile is fed.

## Maintenance rules (read first)

- This file owns the guts of each tile: the integration (local vs cloud + cost flag), entity IDs, data-source/config specifics and setup gotchas, and as-built status. Plus the device/account inventory and the flat entity-reference lookup.
- It does NOT own:
  - A tile's look or interaction — faces, sizes, colour, tap behavior → gui-spec.md.
  - Card mechanics — the renderer/extractor contract, FLIP/packer/layout → engine-spec.md (this file says which entity a tile reads; engine says how the card renders it).
  - Hardware, network, OS/kiosk, the burn-in stack, the feature roadmap/notification policy → family-dashboard-spec.md.
- The join key is the tile name/number, shared with the gui design table. Describe each tile once here (its plumbing) and once in gui (its design) — never restate the other.
- Priority #1 is local/free. Flag every cloud dependency and any recurring cost explicitly. (Today: no recurring fees; several free-but-cloud sources, called out per tile.)
- This is a live snapshot, not a log. No session history, no "we switched from X to Y," no change-log header — write only the current decision and the current status. Regenerate at session close-out: flip planned → built, update entity IDs as they wire in, resolve deferred/blocked items, and delete superseded picks rather than narrating the swap.

## Config architecture — packages

`configuration.yaml` is a thin loader; tile-specific config lives in `packages/<tile>.yaml`, auto-merged at startup via `homeassistant: packages: !include_dir_named packages`. One tile = one file = one place to look — the same separation discipline as the card's four doors. Existing: `packages/weather.yaml`, `markets.yaml`, `sports.yaml`, `home.yaml`, `climate.yaml`, `photos.yaml`, `media.yaml`, and `kiosk.yaml` (the shared kiosk step-out `rest_command`s — not a tile, a cross-tile primitive). (`media.yaml` now exists — it holds the `media_music_night` + `media_all_off` launcher scripts; the TMDB bake automation for Movies is still to add.) (`home.yaml` now holds `input_boolean.notify_mute` + `input_boolean.quiet_hours`, the eight tunable-timing `input_number.wall_*` helpers, the two Data-tab threshold helpers (`wall_hurricane_distance_mi` / `wall_radar_window_min`), and the System Monitor sensor config the Admin System tab reads; `photos.yaml` holds the Useless Facts `rest:` sensor; `kiosk.yaml` now adds `kiosk_display_off`/`kiosk_display_on`/`kiosk_refresh_browser`.) New tiles follow the pattern. (The global file manifest — theme, fonts, the card itself, the radar frame cache — lives in family-dashboard-spec.md.)

Status terms used below: **Live** = running as a real engine face. **Interim** = integration wired + a working button-card tile on the legacy "Mockups" dashboard, but the engine card still renders a placeholder for it. **Planned** = decided, not built. **Deferred / Likely drop** = postponed or probably cut.

**Entity-registry hygiene (a learned gotcha — applies to any tile).** When a sensor's config is wrong at first creation (e.g. the Saints `team_id: NO` → `false`, or a renamed/changed template sensor), HA can leave a stale **restored ghost** holding the clean entity_id, which forces the corrected sensor to register under a `_2` suffix. The clean fix is to **delete the ghost from the entity registry (Settings → Devices & Services → Entities) and let the integration reclaim the clean name** — or rename the live `_2` back to the clean id. **Never edit the config/dashboard/faces to chase the `_2` everywhere**: that bakes the suffix in permanently and resurfaces later as a "this entity is no longer being provided by the integration" orphan when the config shifts again (it cost us a debugging detour on `sensor.iem_cli_*`). Symptoms to watch: an unintended `sensor.<name>_2`, or an entity flagged "no longer being provided."

### 2 — Cameras

- Integration: Reolink first-party HA integration (auto-discovers on LAN; enable RTSP + ONVIF per camera in the Reolink app → Network → Advanced). Local. Four entities live (each camera has a Fluent sub-stream + Clear full-res variant). **The Wall does NOT play these entities directly — it plays go2rtc streams (below);** go2rtc is the broker that pulls the camera RTSP once and fans it out over WebRTC. **NEW HACS dependency: WebRTC Camera (AlexxIT) — the `custom:webrtc-camera` card.**
- **Stream tech (AS BUILT — go2rtc + WebRTC, LIVE).** go2rtc is stood up: `/config/go2rtc.yaml` defines four streams from the Reolink RTSP URLs and PRELOADS them so each stays warm 24/7. HA's bundled go2rtc (the managed `go2rtc` integration on HA OS) reads that file at startup — we declare ONLY `streams:` + `preload:` and let HA own all ports/ICE (do NOT add a `webrtc:`/`api:` block — that fights the managed setup and breaks the integration load). The card side is the **WebRTC Camera** HACS integration (`custom:webrtc-camera`, AlexxIT) + integration added with the **go2rtc URL left blank** (uses HA's built-in server). The faces point the card at the **go2rtc stream NAME** (not the HA camera entity) with `mode:"webrtc,mse"` (WebRTC first, MSE fallback — MSE always connects on the LAN). The Reolink HA camera ENTITIES are NOT used by the Wall (they stayed `supported_features:2`/HLS and don't auto-upgrade — we bypass them entirely by playing the go2rtc stream).
- **go2rtc.yaml stream map:** `driveway_clear` → `rtsp://…@>>>[CAM1_IP]<<<:554/h264Preview_01_main` · `driveway_fluent` → `…_sub` · `front_door_clear` → `rtsp://…@>>>[CAM2_IP]<<<:554/h264Preview_01_main` · `front_door_fluent` → `…_sub`. (Credentials live in the file, in private `/config`, not web-served — same trust model as `markets_history.py`.) **Only the two `_clear` streams are actually used** (see below); the two `_fluent` preloads are dead weight now and a candidate to trim next session.
- **Streams used = the two Clear/main, at EVERY size.** The old Fluent-on-1×1 / Clear-on-larger split was DROPPED: on the wired LAN the Fluent bandwidth saving is meaningless, go2rtc preloads fewer streams, and one sharp stream beats a soft-upscaled sub-stream at 4×3. So 1×1, 2×2 quad, and 4×3 all play `driveway_clear` / `front_door_clear`.
- **Cold-start CURED — persistent video layer (AS BUILT; engine-spec.md).** The webrtc-camera cards never live in the face anymore. The face renders only transparent `.cam-window` markers (each carrying `data-cam-stream` / `data-cam-tag`); the engine owns ONE persistent per-tile video layer (`.cam-video-layer`, a stable sibling of `.tile-frame`), builds each go2rtc stream's card once (`loadCardHelpers`), keeps it warm + decoding, and REPOSITIONS the matching window over the current face's markers at every motion-settle. So an open/flip/grow/reflow MOVES an already-decoding window instead of rebuilding it — no black beat. Camera faces opt in with `video:true` + `videoConfig(stream)`; the engine stays tile-agnostic (FACE_VIDEO / FACE_VIDEO_CFG). Engine mechanics in engine-spec.md → Persistent video layer. NOT a tiles/wiring change.
- Reolink camera entities (live, but NOT what the Wall plays): `camera.driveway_fluent` / `camera.driveway_clear` (driveway, IP >>>[CAM1_IP]<<<); `camera.front_door_fluent` / `camera.front_door_clear` (front door, IP >>>[CAM2_IP]<<<). go2rtc pulls the same RTSP the integration uses; the Wall plays the go2rtc stream by name.
- **2×2 bottom half — the daily activity pair (BUILT, LIVE).** Top two quadrants stay the live Clear feeds (driveway + front door). The bottom two are a self-contained "what happened" pair built on a snapshot store WE own:
  - **The event store (the key architectural call).** The Reolink integration gives AI detections as live BINARY SENSORS (audited live — **driveway: person / vehicle / pet** · **front door: person / visitor** [the doorbell press]; raw `motion` exists on both but is excluded; **no package/animal on these models**), NOT as tagged clips — the recordings live on the camera SD card reachable only through HA's media browser, which is untagged, browse-only, and not queryable as a list. So we DON'T fight the media browser. Instead: when an AI binary sensor fires, an HA automation (`packages/cameras.yaml`, mode: restart, trigger ids per sensor; front-door triggers fold to `person`) grabs a still snapshot (`camera.snapshot`) from that camera ~2s after the trigger (lands nearer the middle of the action than the moment of detection) and saves it to `/config/www/snapshots/{utcnow %Y%m%dT%H%M%SZ}_{type}_{camera}.jpg`. This is the tagged, queryable event store Reolink won't give us — local, tiny JPEGs, no decode. Both bottom quadrants read the manifest a stdlib-only `snapshots_index.py` re-bakes after each capture (the radar/markets bake→read-local pattern); `events.json` keeps a newest-first list, the face fetches it through the canonical extractor (`quadExtractor`, 30 s throttle, token-bumped on fresh).
  - **Bottom-left — event reel (LIVE):** cycles the snapshot JPEGs newest-first, rAF crossfade every 4 s, self-cleaning (stops on `!el.isConnected`). TAP THE REEL → opens the 4×3 snapshot **viewer** face (below). Source-agnostic by design: if Frigate is ever added it becomes a richer backing source (real clips/scrubbing) with no face rewrite.
  - **Bottom-right — event list (LIVE):** the SCROLLABLE chronological index of the same folder (type · camera · time). Full list, not capped-N — an early-morning trigger must not be buried under the afternoon's activity, so the user can scroll the whole window. Uses the canonical **ScrollList** component (built by Calendar; engine-spec.md). Header shows total count + the three filter chips.
  - **Filter chips (Person / Vehicle / Pet) drive BOTH:** single-select (tap-again clears, no stacking), reel + list filter together, viewer index also resets to the newest of the new set.
  - **Coupling:** tapping a list row drives the reel to that event's snapshot, then **pauses the reel's auto-cycle for 30 s** before resuming. While idle (no tap) the reel auto-cycles. List indexes, reel plays — a "browse the day" pair.
  - **Empty state:** a clean "No activity" face (or "No <filter> events" when filtered) for the genuinely-no-events window (e.g. a quiet overnight stretch).
  - **Retention = rolling 24 hours.** A trailing-24h sliding window: an HA automation calls `snapshots_index.py` which prunes the folder of anything older than 24 h, oldest snapshots dropping continuously as the window advances. A hard 300-file cap is the backstop against a pathologically busy day. Chosen over a calendar-day wipe because a midnight wipe erases overnight/late-evening events before they're ever seen (bed at 9–10 PM → that night's activity gone by morning); the rolling window guarantees the morning-after check always works. Single span, so no day dividers. File cost trivial (JPEGs).
- **4×3 snapshot viewer — `cameraSnapshots` (BUILT, LIVE).** The big-still review face, reached ONLY by tapping the 2×2 reel (the engine's `swipeExclude` flag walls it off the live-feed flip cycle, so a swipe between feeds at 4×3 never lands on it). Layout: a big still on the LEFT (~74%, `object-fit: contain`) + a narrow event RAIL on the right (~26%) with the same filter chips + ScrollList of events (the SHOWN row is marked with a left-edge accent + lit background). The viewer **owns its own horizontal drag** as a face-local two-layer translateX slide that mirrors the 1×1 live-feed feel verbatim (drag-follow, 36 px commit threshold, springs back if short, wraps the filtered list, click-swallow on release — see engine-spec.md → swipe-exclude). A rail row tap (`view-to`) jumps the big image to that event (instant, no animation). A filter chip tap resets the viewer to the newest of the new filter. The frame band steps back to 2×2 like any open tile. Stills only (no decode) — the viewer is for REVIEW; the live feeds have their own 4×3 full-screen faces. (Real CLIP playback is the deferred Frigate upgrade.)
- Person/vehicle detection — Frigate dropped for v1, and the activity pair above is now fully DECOUPLED from it (self-contained snapshot store). Re-adding Frigate later is optional and only upgrades the reel's source to real clips. Frigate is also hardware-gated: it needs the box's iGPU via OpenVINO and CANNOT meaningfully run on the emulated VM — so it's a Phase-C-on-hardware task by necessity, never a VM task.
